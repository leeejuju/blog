多 Agent 系统里有一个比较常见的问题：**Agent，尤其是类似 ChatGPT, Gemini, 这类 web 端的 Chatbot在执行期间，会出现这样一个问题，即**

本文记录一种解法——队列分离 + 粘性执行。

---

## 问题

之前做 Multi Search Agent 的时候踩过一个坑：一个简单的用户查询，触发了多个 Web Search 链路，好几个 sub-agent 在干差不多的事，token 烧了，响应也慢了。

回溯原因：

1. **任务路由没有明确边界**——Leader 分解出来的子任务，描述太模糊，谁都可以抢。
2. **执行没有粘性**——同一类子任务每次被不同 agent 认领，上一个 agent 积累的工具调用经验、搜索结果上下文，下一个完全用不上。

---

## 思路

把任务按类型拆进独立队列，每个队列绑定一个固定 Agent：

```text
用户查询
  │
  ▼
Leader Agent（分解 + 路由）
  │
  ├── 队列 A: Web 检索   →  Agent A（只做 Web 检索）
  ├── 队列 B: 代码检索   →  Agent B（只做代码检索）
  └── 队列 C: 数据查询   →  Agent C（只做结构化数据查询）
```

### 队列分离

- 一个队列只做一件事，Agent 不跨队取任务。
- 任务描述必须具体到可执行级别——"research semiconductor shortage" 不行，得拆成"查 2025 Q1 全球半导体产能数据"这种粒度。
- 队列数随查询复杂度缩放：简单事实查询 1 个队列够用，综合研究可以开到 10+ 个队列。

### 粘性执行

同一个 Agent 持续处理同一类任务，直到该队列清空：

- **上下文复用**——前面几次工具调用的结果，后面可以直接参考，不用每次从头来。
- **调度逻辑简化**——不用反复决策"这次谁来干"，队列即分配。
- **行为可追踪**——每个 Agent 的执行轨迹固定在其队列范围，调试时不用在多个 agent 的日志里跳来跳去。

粘性执行的前提是**清晰的任务边界**——Agent 得明确知道自己能干什么、不能干什么、输出什么格式。这本质上是 Prompt Engineering 的问题。

---

## 工程实现

尝试过三种方式，最终选了 LangGraph Workflow：

### 节点插入（不适用）

`compiled_graph` 返回的是 Compiled Graph，编译后无法再插入新节点。队列需要动态扩展时行不通。

### Middleware 协程（笨重）

在树状上下游设多个 Tools，用协程驱动多个 Observation。能跑，但代码复杂度高，调试困难。

### LangGraph Workflow + MapReduce（最终方案）

```text
Leader
  │
  ├── fan-out ──→ [队列A] [队列B] [队列C]
  │                 │       │       │
  │                 ▼       ▼       ▼
  │               AgentA  AgentB  AgentC
  │                 │       │       │
  │                 └───────┼───────┘
  │                         ▼
  └── fan-in ←─── 结果归并
  │
  ▼
判断是否继续检索
  │
  ├── 不足 → 回到 fan-out
  └── 充足 → 输出
```

几个关键决策：

- **Sub-agent 无状态**——只管执行，结果总结好反馈给 Leader。LangChain 文档也提过这点。
- **模型选型下放**——Sub-agent 直接用 Flash 级别模型（如 Haiku），执行效率优先。推理和判断交给 Leader 的强模型。
- **队列粘性在 Graph 层面体现**——每个 Agent 节点只消费自己对应队列的输出，不跨队取任务。

---

## 不太重要的结尾

队列分离和粘性执行不是死规则，是一种启发式——让 Agent 知道自己该干什么、不该干什么，然后持续干好它该干的事。

与其写一堆禁止性规则，不如把正向边界定义清楚。
